# Ревью проекта (2026-06-12)

Полное ревью кодовой базы: бизнес-логика, тесты/качество, инфраструктура и безопасность.
Состояние на коммит `991f4e7`. Тесты: **68/68 зелёные** (при подставленном env), логика расчёта
таблиц и сетки в основных сценариях корректна, данные ЧМ-2026 (48 команд, 12 групп,
сетка 73–104, слоты третьих мест) сверены с `docs/tournament-format.md` — расхождений нет.

## 🔴 Критично

### 1. «Протухшие» инлайн-кнопки портят данные чужого матча
Callback-данные кнопок счёта (`gs:2:1`, `ps:2:1`) и победителя (`bwin:H`/`bwin:A`)
не содержат номер матча — хэндлер берёт текущий матч из FSM:
- `src/keyboards/common.py:21-35, 94-100`
- `src/handlers/group_stage.py:118-128`
- `src/handlers/playoff.py:132-142, 156-181`

Сценарии порчи данных:
1. После «✍️ Свой счёт» + ввода текстом клавиатура на старом сообщении остаётся.
   Тап по старой кнопке «2:1» позже молча сохраняет 2:1 в **текущий** матч.
2. `winner_keyboard` после выбора никогда не убирается. При следующей ничьей тап по
   старой кнопке «✅ Команда» применит сторону H/A к новому матчу — дальше пройдёт
   не та команда, что написана на кнопке.

**Фикс:** зашивать номер матча в callback_data и сверять с FSM; убирать клавиатуру
у сообщения с ничьей (`edit_reply_markup(None)`).

### 2. Postgres опубликован на 0.0.0.0 с дефолтным паролем
`docker-compose.yml:6,9`: `"${POSTGRES_PORT:-5432}:5432"` биндит на все интерфейсы,
пароль по умолчанию `change-me`. Если dev-compose поднять на VPS без файрвола —
база доступна из интернета с известным паролем.

**Фикс:** `127.0.0.1:${POSTGRES_PORT:-5432}:5432` или убрать `ports` совсем
(бот ходит по compose-сети); убрать дефолт пароля — пусть compose падает без
заданного `POSTGRES_PASSWORD`.

### 3. «Золотая бутса»: число голов теряется навсегда
`src/handlers/awards.py:198-207`: `upsert_award(TOP_SCORER, player_id=...)` коммитится
**до** вопроса «Сколько голов?». Если пользователь бросил бот на этом шаге (или бот
перезапустился — FSM в `MemoryStorage`), при возврате шаг пропускается:
`_first_incomplete_step` (`awards.py:33-43`) проверяет только наличие награды,
а не заполненность `int_value`. Исправить можно только полным `/reset`.

**Фикс:** сохранять награду только после ввода голов, либо проверять
`int_value is not None` в `_first_incomplete_step`.

### 4. `pytest` падает после чистого клона
Нет `tests/conftest.py`, а `src/config.py:34` создаёт `settings = Settings()` при
импорте — collection падает с `ValidationError: BOT_TOKEN, DATABASE_URL required`.

**Фикс:** `tests/conftest.py` с `os.environ.setdefault(...)` до импорта `src.config`,
либо ленивый `get_settings()` с `@lru_cache`.

## 🟡 Средние проблемы

### Бизнес-логика
- **Тай-брейк личных встреч не реприменяется к подгруппе** (`src/services/standings.py:114-136`).
  При равенстве в мини-таблице решает имя команды, а по регламенту FIFA критерии должны
  заново применяться к оставшимся равными. Контрпример: цикл A>B>C>A — B и C равны в
  мини-таблице трёх, но по FIFA B выше (личная встреча); код отсортирует по алфавиту.
  Существующий тест `test_head_to_head_breaks_equal_points_and_gd` ветку тай-брейка
  на самом деле не покрывает (в нём очки разные: 3 vs 0).
- **Старая кнопка «Начать прогноз» сбрасывает FSM посреди плей-оффа**
  (`src/handlers/start.py:36-47` → `group_stage.send_next_match:48-57`): при заполненных
  72 матчах выполняется `state.clear()` — пользователь выбивается из `Playoff.*`/`Awards.*`.
- **Race conditions select-then-insert** (`src/db/repo.py:27-37, 115-139, 220-248, 322-348`):
  при конкурентных апдейтах уникальные констрейнты спасают данные, но пользователь получает
  «⚠️ Что-то пошло не так». Лечится `ON CONFLICT DO UPDATE`.
- **N+1 на каждый клик в плей-оффе** (`src/services/playoff.py:22-42, 53-80`):
  ~50–80 запросов на каждый введённый счёт (пересчёт 12 групп + SELECT на каждый матч сетки).
  Имена игроков в `report_data.py:126,135` и `awards.py:277,286` — одиночными запросами в цикле.
- **Третьи места не по официальной таблице FIFA** (`src/services/bracket.py:40-75`):
  бэктрекинг вместо таблицы 495 комбинаций — ограничения слотов соблюдены, но конкретные
  пары R32 могут отличаться от назначений FIFA. Фолбэк (строки 72–75) молча нарушает
  ограничения — нужен хотя бы лог ошибки.
- **Колбэки без хэндлера — вечный «спиннер»** (`src/handlers/__init__.py`): fallback ловит
  только `message`. Нужен catch-all `@router.callback_query()` с `call.answer(...)`.

### Инфраструктура
- `.env.example:8,11`: `localhost` как хост БД ломает dev-compose (внутри контейнера
  `localhost` ≠ сервис `db`). Нужны два примера или комментарий dev/prod.
- `src/bot.py:46-58`: нет `await engine.dispose()` после остановки polling
  (`try/finally` вокруг `start_polling`).
- `src/bot.py:37`: `MemoryStorage` — при каждом redeploy все пользователи в середине
  сценария теряют FSM-состояние. Минимум — зафиксировать как осознанное решение,
  лучше — RedisStorage.
- `src/db/session.py:7`: пул не настроен явно (дефолт до 15 коннектов) — на общем VPS
  с чужим Postgres стоит задать `pool_size`/`max_overflow`/`pool_timeout`.
- `docker-compose.prod.yml:21`: `network_mode: host` — бот видит все локальные сервисы
  VPS; безопаснее compose-сеть + `extra_hosts`. В проде нет `pids_limit`.
- Версии не зафиксированы: `postgres:16` (минор), `python:3.12-slim`, зависимости через
  `>=` без lock-файла — сборка невоспроизводима.
- `migrations/versions/677311f11ba0_initial_schema.py:41`: миграция автогенерирована
  против SQLite (`(CURRENT_TIMESTAMP)` вместо `now()`) — будущие autogenerate против
  Postgres могут давать ложные диффы. По составу моделям соответствует.

### Качество / сборка
- `pyproject.toml`: `packages = ["src"]` — в wheel/sdist не попадут подпакеты
  (`src.handlers`, `src.services`, ...). Editable-install это маскирует.
  Нужно `[tool.setuptools.packages.find] include = ["src*"]`.
- `ruff format --check .` — 26 из 55 файлов не отформатированы (нарушает правило
  CLAUDE.md); `ruff check` — 1 ошибка `I001` в `scripts/loadtest.py:12`.
  Ruff не закреплён по версии (`>=0.6`, стоит 0.15.x) — формат «плывёт».
- Пробелы покрытия (по приоритету):
  1. Тай-брейк h2h при реально равных очках; круговое равенство 3 команд.
  2. Граница 8/9 при отборе третьих мест (равенство показателей у 8-го и 9-го).
  3. Ничья в плей-офф → выбор победителя (`waiting_score → waiting_winner`) — не покрыто вовсе.
  4. Редактирование группового матча после старта плей-оффа: `upsert_bracket_teams`
     не перезаписывает решённые матчи → в сетке остаются устаревшие команды
     (семантика нигде не зафиксирована).
