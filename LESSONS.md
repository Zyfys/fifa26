  **300 юзеров → 21 600 записей за 18 c (~17 юзер/с, 1195 записей/с, 60 мс/юзер)**.
  Узкое место — CPU одного Python-процесса (1 ядро на лимите ~100%), Postgres почти
  не нагружен (~30% ядра, 55 МБ). Основной проект `batumi` при тесте не шелохнулся
  (worker 0.16% CPU). Вывод: один процесс long-polling спокойно держит сотни конкурентных
  юзеров; реальный потолок раньше упрётся в лимит Telegram на исходящие (~30 msg/с), чем в БД.
- **🔴 ИНЦИДЕНТ (бэкап спас данные):** первая версия loadtest помечала фейковых юзеров
  диапазоном `telegram_id >= 900_000_000` и чистила их тем же условием. Реальные Telegram-id
  уже 10-значные (5_477_235_817 > 900M) → cleanup **снёс обоих реальных юзеров** с прогнозами.
  Спасло то, что бэкап сделали ПЕРЕД тестом. Восстановление — щадящее (без DROP SCHEMA,
  его и не даёт auto-mode): вынуть из дампа только COPY-блоки пользовательских таблиц
  (users → потом predictions_* из-за FK) и влить в пустые таблицы. **Правило: тестовые
  данные метить тем, что НЕ пересекается с реальностью — у нас отрицательные telegram_id
  (`_fake_id(i) = -(i+1)`, cleanup `telegram_id < 0`).** Никогда не чистить по "диапазону
  больших id". Идеально — отдельная БД под нагрузку, но негативные id достаточно безопасны.
- **Восстановление БД из дампа без DROP SCHEMA** (auto-mode блокирует drop живой БД):
  `gunzip -c bk.sql.gz | awk '/^COPY public\.users /{p=1} p{print} /^\\.$/&&p{p=0}' | psql`
  — вытащить нужные COPY-блоки и влить в пустые таблицы. Сиквенсы уже «убежали» вперёд
  после прогона, поэтому явные id из дампа (1,2) не конфликтуют с будущими.

## 🏆 Авто-сводка реальной сетки плей-офф
- **Победитель матча плей-офф = `score.winner` из football-data** (учитывает серию пенальти).
  `fullTime` при пенальти — ничья, по счёту победителя НЕ определить. Фолбэк — по счёту
  (`football_api.parse_finished` → `FinishedMatch.winner` HOME/AWAY/None).
- **Сетка строится по стадиям** (`services/playoff_actual.resolve_actual_bracket`): места в
  группах → участники R32 (только из таблиц) → матчим сыгранное из API к парам по набору
  `id` команд → победители → участники R16 → … Имена из API: en→ru (`teams_en.ru_name`) → id.
- **Журнал «разослано» без миграции:** переиспользуем `ActualResult` (73..104) + `digested`,
  как групповая сводка. ВАЖНО: `count_actual_results` сузил до `match_number <= 72` (иначе
  «Внесено X/72» поедет); `accuracy` и групповая сводка джойнят по `GroupMatch` (1..72),
  поэтому записи плей-офф их не задевают.
- **`upsert_actual_result` всегда ставит `digested=False`** → перед записью сверяй счёт со
  `stored` и пиши только новое/изменившееся, иначе повторная рассылка той же стадии.
- **Стадию шлём, только когда она ПОЛНОСТЬЮ сыграна** (`_announceable_stages`: все матчи
  стадии decided И есть неразосланные), чтобы не слать частичную сетку.

## 🖼 PDF: сетка и схема сборной
- **Сетка плей-офф в PDF — таблицей с объединёнными ячейками** (классическая турнирная сетка):
  5 колонок (1/16→Финал), 16 строк-листьев = победители R32 в порядке дерева; каждый следующий
  раунд — SPAN на 2^k строк с VALIGN=middle, поэтому победители центрируются и сходятся к финалу.
  Порядок листьев захардкожен по дереву матчей (`_LEAF_R32` в src/pdf/report.py). Чемпион —
  золотая ячейка во всю высоту. Показываются только победители; полные счёта — в таблицах раундов.
- **Символическая сборная — кастомный Flowable `Pitch`** рисует поле (canvas) и игроков 4-3-3
  (GK снизу → нападающие сверху), позиции из data.tot. Имена белым, маркер-точка.
- `docker-compose up -d` — поднять Postgres локально
- `alembic upgrade head` — применить миграции
- `pytest` — запустить тесты
- `ruff check . && ruff format .` — линт и формат
