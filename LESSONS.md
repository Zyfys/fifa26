# Уроки и заметки

## 💡 Что узнал
- **Заявки ЧМ-2026 опубликованы** (по 26 игроков, сданы в FIFA к 01.06.2026, объявлены 02.06.2026).
  Турнир стартует 11.06.2026. База игроков собираема — главный риск снят.
- **Источники данных игроков:**
  - Полные ростеры с позициями: ESPN — https://www.espn.com/soccer/story/_/id/48757621
    и NBC Sports — https://www.nbcsports.com/soccer/news/2026-world-cup-squads-confirmed-rosters-for-all-48-teams
  - Возраст / дата рождения: Wikipedia (страницы заявок сборных).
  - Объём: 48 команд × 26 = ~1248 игроков.
- **Стратегия сида игроков:** собрать через WebFetch в промежуточный JSON
  (team, name, position GK/DF/MF/FW, birth_date), затем грузить в БД сид-скриптом.
  Данные проверять — заявки могли меняться из-за травм.
- **Источник составов:** страница `2026_FIFA_World_Cup_squads`. HTML через WebFetch
  обрезается после ~1.5 групп; «сырой» wikitext (`?action=raw`) добивает до ~группы D.
  Для групп E–L использовать секционный API Wikipedia (по одной секции команды).
  Запрашивать не более ~2 команд за раз и проверять, что в составе 26 игроков.
- **Транслитерация имён — к ревью:** ~1248 имён переведены вручную, возможны неточности
  (особенно корейские/арабские/балканские). Свериться перед продакшеном.
- **3 команды по 25 игроков** (Канада, Австрия, Иордания) — источник отдал на 1 меньше 26.
  Дозагрузить недостающих перед продакшеном. Всего в БД 1245 игроков (48×26 − 3).
- **Локальная разработка БЕЗ Docker:** `DATABASE_URL=sqlite+aiosqlite:///./fifa26_dev.db`,
  `BOT_TOKEN=dummy`, затем `py -m alembic upgrade head` + `py -m src.seed`. Проверено
  end-to-end: 12 групп/48 команд/72 матча/32 слота сетки/1245 игроков. Postgres+Docker — позже.
- **Запуск бота локально:** `DATABASE_URL=sqlite+aiosqlite:///./fifa26_dev.db py -m src.bot`
  (токен берётся из .env). PYTHONIOENCODING=utf-8 — иначе кириллица в логах падает на cp1251.
- **FSM на MemoryStorage** — dev-вариант (сбрасывается при перезапуске). Прогноз хранится в БД,
  поэтому прогресс не теряется: следующий матч вычисляется из БД. Для прода — RedisStorage.
- **Привязка 3-х мест к слотам R32 — приближение, не офиц. таблица FIFA (495 комбинаций).**
  Реализовано через паросочетание с возвратом ([src/services/bracket.py](src/services/bracket.py)):
  каждый слот получает 3-е место из разрешённого набора групп, все 8 распределяются без
  конфликтов. Тест перебирает все 495 комбинаций — валидное назначение есть для каждой.
  Сетка играбельна и корректна по ограничениям; для 100% совпадения с FIFA подложить офиц. таблицу.
- **Плей-офф: ничья → выбор прошедшего** (серия пенальти) кнопками. Победитель/проигравший
  каждого матча резолвятся в следующий раунд функцией playoff.resolve_bracket (вызывается
  после каждого прогноза). Чемпион = победитель матча 104.
- **Фаза 5 (PDF):** выбран **ReportLab** (не WeasyPrint) — чистый Python, без нативных
  зависимостей, ставится pip'ом на Windows. Кириллица через TTF: автоопределение шрифта
  (Arial на Windows, DejaVuSans в Docker — ставится `fonts-dejavu-core` в Dockerfile).
  Рендер в отдельном потоке (`asyncio.to_thread`), чтобы не блокировать бот. Эмодзи в PDF
  не используем (нет глифов в шрифте). Проверка PDF — инструментом Read (читает страницы).
- **Фаза 4 (награды/сборная):** выбор игрока из ~1245 — через drill-down «команда → игрок»
  с пагинацией (12 на страницу). Фильтры в SQL: позиция (вратарь), возраст (молодой игрок —
  2005 г.р. и младше, [src/services/awards.py](src/services/awards.py)). Чемпион/финалист/3-е
  место НЕ переспрашиваем — берём из сетки. В сборной игрок не повторяется (исключаем выбранных).
  Шаги резюмируются из БД (первый незаполненный) — прогресс не теряется.
- **Проверка без Docker:** локально стоит Python 3.14 (`py`) — можно `py -m pip install`
  sqlalchemy/pytest и гонять тесты схемы на SQLite (`tests/test_schema.py`) и данных,
  не дожидаясь Postgres. Полную миграцию/async всё равно проверять в Docker.
- **Фаза 6 (харденинг):** глобальный error-handler через `dp.errors.register(on_error)`
  ([src/bot.py](src/bot.py)) — логирует трейсбэк и дружелюбно отвечает пользователю,
  само уведомление в try/except (не должно ронять обработчик ошибок). В колбэках
  `call.message` бывает None / `InaccessibleMessage` (старое сообщение) → helper
  `accessible_message(call)` ([src/handlers/callbacks.py](src/handlers/callbacks.py)):
  при недоступности шлёт алерт «устарело, /start» и возвращает None — хэндлер выходит.
  Валидация счёта вынесена в один модуль `services/scores.py` (`parse_score` /
  `score_error_message`, формат «2:1»/«4-4», MAX_SCORE=30) — используется и в группах,
  и в плей-офф. Разделитель в regex включает `\s`, поэтому «2 1» парсится, «21» — нет.
- **Ruff и миграции:** автогенерённые Alembic-миграции линтить смысла нет (формат задаёт
  Alembic) — в pyproject `extend-exclude = ["migrations/versions"]`. Enum'ы предметной
  области — `enum.StrEnum` (а не `str, enum.Enum`): UP042, поведение идентично.
- **Фаза 7 (безопасность):** Dockerfile делал `COPY . .` БЕЗ `.dockerignore` → `.env`
  (токен+пароль) и `*.db` запекались в слои образа. Обязателен `.dockerignore` (исключить
  `.env*`, `*.db`, `.git`, тесты, `*.md`). Контейнер — от non-root (`useradd appuser` + `USER`).
  В PDF (ReportLab `Paragraph`) пользовательские данные экранировать `xml.sax.saxutils.escape`
  (Paragraph парсит мини-XML). На общем VPS: `chmod 600 .env`, лимиты `mem_limit`/`cpus`.
- **Анти-флуд — два уровня (не конфликтуют):** (1) транспортный `ThrottlingMiddleware`
  ([src/middlewares.py](src/middlewares.py)) — `dict[user_id]->last_ts`, интервал 0.5 c,
  регистрируется `outer_middleware` ДО `DbSessionMiddleware`, чтобы дроп шёл до открытия
  сессии БД; для outer-middleware на `dp.update` user_id достаётся из `Update` вручную
  (нет ещё `event_from_user`). (2) доменный кулдаун «3 попытки → пауза»
  ([src/services/antiflood.py](src/services/antiflood.py), чистая логика поверх FSM-данных:
  `bad_attempts`/`cooldown_until` через `time.monotonic()`), подключается общим хелпером
  `guard_text_input` ([src/handlers/callbacks.py](src/handlers/callbacks.py)) в трёх
  текстовых хэндлерах (группы/плей-офф/награды). Кулдаун — только на текст, кнопки счёта
  работают. MemoryStorage: счётчики обнуляются при рестарте — приемлемо.
- **Catch-all фоллбэк:** `@router.message()` без фильтра в роутере, включённом ПОСЛЕДНИМ
  ([src/handlers/fallback.py](src/handlers/fallback.py)) — ловит мусорный текст вне нужных
  состояний (не лезет в БД), не мешая основным хэндлерам (порядок роутеров важен).
- **Деплой на общий VPS (изоляция):** отдельная БД+роль на существующем Postgres,
  роль = владелец БД и схемы (PG 15+: `ALTER SCHEMA public OWNER TO ...`, иначе Alembic не
  создаст таблицы), `REVOKE CONNECT ... FROM PUBLIC`. Отдельная папка `/opt/fifa26/`,
  `name: fifa26` + `container_name` в compose (своё пространство имён, без конфликта с
  основным проектом). Бэкап только своей базы `pg_dump -d fifa26` (НЕ `pg_dumpall`).
  Полный runbook — [docs/DEPLOY.md](docs/DEPLOY.md). Alembic тут на asyncpg (migrations/env.py
  через `async_engine_from_config` + `run_sync`) — sync-драйвер/psycopg2 НЕ нужен.

## 🖼 PDF: сетка и схема сборной
- **Сетка плей-офф в PDF — таблицей с объединёнными ячейками** (классическая турнирная сетка):
  5 колонок (1/16→Финал), 16 строк-листьев = победители R32 в порядке дерева; каждый следующий
  раунд — SPAN на 2^k строк с VALIGN=middle, поэтому победители центрируются и сходятся к финалу.
  Порядок листьев захардкожен по дереву матчей (`_LEAF_R32` в src/pdf/report.py). Чемпион —
  золотая ячейка во всю высоту. Показываются только победители; полные счёта — в таблицах раундов.
- **Символическая сборная — кастомный Flowable `Pitch`** рисует поле (canvas) и игроков 4-3-3
  (GK снизу → нападающие сверху), позиции из data.tot. Имена белым, маркер-точка.
- `docker-compose up -d` — поднять Postgres локально
- `alembic upgrade head` — применить миграции
- `pytest` — запустить тесты
- `ruff check . && ruff format .` — линт и формат
